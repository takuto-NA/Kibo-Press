# Kibo-Press 要件整理

## 1. プロダクト概要

**Kibo-Press** は、Markdown 文書を、報告書・提案書・技術文書のような整った見た目の PDF に変換する VS Code 拡張である。

利用者は VS Code 上で Markdown を執筆し、コマンドひとつで、高品質なレイアウトを持つ PDF を生成できる。出力された PDF は、単なる印刷プレビューではなく、Word やドキュメントソフトで丁寧に整えたような紙面品質を目指す。

## 2. 解決したい課題

一般的な Markdown → PDF 変換には、次の問題がある。

- 見た目が簡素で、文書としての完成度が低い
- 余白、見出し、表、画像、コードブロックなどの見た目が雑になりやすい
- 提案書、報告書、配布資料としてそのまま使いにくい
- VS Code 内で完結して気持ちよく運用しにくい

Kibo-Press はこれを解消し、Markdown の軽快さを保ちながら、提出・共有に耐える PDF を生成する。

## 3. 想定ユーザー

### 主対象

- VS Code で文書を書く開発者
- Markdown で技術文書や社内文書を書く人
- 提案書、報告書、仕様書を素早く整形したい人

### 典型ユースケース

- 技術メモを体裁の整った PDF にする
- 提案書やレポートを Markdown で作成し、そのまま提出用 PDF にする
- 社内ドキュメントをテンプレート付きで統一フォーマット化する
- エンジニアが Word を使わずに見栄えのよい文書を作る

## 4. 提供価値

### 4.1 美しい紙面

- 余白、行間、見出し階層、表、画像、コードブロックを整える
- 「文書として読める」「提出できる」見た目を実現する

### 4.2 Markdownのまま書ける

- 執筆体験は Markdown の軽さを維持する
- 特殊な記法を強要しない

### 4.3 VS Codeで完結する

- プレビュー、PDF出力、テーマ切り替えを拡張内で扱える
- CLI や外部ツールを毎回意識しなくても使える設計にする

## 5. MVP で固定した仕様（実装準拠）

### 5.1 変換パイプライン

`Markdown → Pandoc（typst 出力）→ Typst compile → PDF`

### 5.2 設定の責務

| ソース | 用途 |
|--------|------|
| Markdown front matter | ドキュメント単位（タイトル、著者、目次、テーマ名など） |
| VS Code `kiboPress.*` 設定 | 既定テーマ・出力先・ツールパス・エクスポート後の自動オープンなど（文書メタデータの注入は MVP では未実装） |
| `themes/<name>/theme.yaml` | テーマ既定（紙面・フォント名など） |

MVP では **`kibo-press.yaml` は導入しない**。

利用者向けのキー一覧・体裁の変え方は **[カスタマイズ](customization.md)** を正とする。

### 5.3 設定マージ優先順位

1. front matter  
2. ワークスペース設定  
3. ユーザー設定  
4. テーマ `theme.yaml` 既定  

実装の詳細と MVP での差分（workspace / user からのメタデータ注入は現状空）は [アーキテクチャの設定マージ](architecture.md#4-設定マージ) を参照する。

### 5.4 Open Preview

- **最後に生成した PDF を OS の既定ビューアで開く**（MVP）。
- `Export PDF` 成功時に通知から「開く」を選択可能。
- `kiboPress.openPdfAfterExport` でエクスポート後の自動オープンを切り替え。

### 5.5 推奨日本語フォント（紙面品質）

Typst で次を既定候補とする（未インストール時は README / architecture のフォールバック方針に従う）。

- 本文・見出し: Noto Serif CJK JP  
- 表・キャプション・補助: Noto Sans CJK JP  
- コード: Noto Sans Mono  

## 6. 機能要件（MVP）

### A. Markdown から PDF を出力

- 現在開いている `.md` を PDF に変換
- 出力先を指定可能
- コマンドパレットから実行

### B. 文書テーマ

- 少なくとも `business-report` テーマ
- ページ、余白、フォント、見出し、段落、表、コード、ヘッダ／フッタ、ページ番号を制御

### C. front matter / VS Code 設定

- タイトル、著者、日付、サブタイトル、会社名、目次、テーマ名、表紙有無 など（実装で順次対応）

### D. VS Code コマンド

- `Kibo-Press: Export PDF`
- `Kibo-Press: Export PDF with Theme`
- `Kibo-Press: Open Preview`
- `Kibo-Press: Create Document Config`（front matter 雛形挿入）

### E. エラーハンドリング

- Pandoc / Typst 未検出、変換失敗、不正な front matter を通知・Output Channel に記録

## 7. 非機能要件

- 再現性: 同一入力・同一ツール版・同一フォント前提で安定した版面
- パフォーマンス: 数ページは実用的な待ち時間
- 保守性: テーマと拡張コードの分離、変換はサービス／パイプライン層に集約

## 8. 成功条件（MVP）

- VS Code 上で Markdown を 1 コマンドで PDF 出力できる
- 共有してよいと感じられる紙面（見出し、本文、表、画像、コードが整う）
- 利用者が毎回外部 CLI を手打ちしなくてよい

## 9. スコープ外（初期）

- Word 完全互換レイアウト、WYSIWYG、高度な組版 GUI、クラウド同期など
